Recommendation System Literature Review for Algorithmic Problems

This report explains what each referenced paper or source contributes and why it is useful for building a recommendation system for algorithmic problems. The target system is the one we discussed earlier: a platform that estimates problem difficulty, models user skill by topic and subtopic, and recommends the next problem at an appropriate challenge level.

What We Are Trying to Build

We want a system that does three things well. First, it should estimate a user's ability across topics such as arrays, graphs, recursion, dynamic programming, and subskills such as BFS or sliding window. Second, it should estimate the difficulty of each problem, ideally using both expert tags and interaction data from many learners. Third, it should recommend problems that are neither too easy nor too frustrating, while also helping the learner improve in weaker areas.

The literature below is useful because each source addresses one or more of these parts: online rating, latent difficulty estimation, mastery tracking over time, programming-specific recommendation, or the statistical weaknesses of naïve rating systems.

Paper-by-Paper Review

1. Pelanek 2016

Pelanek, R. (2016). Applications of the Elo rating system in adaptive educational systems. Computers & Education, 98, 169-179. DOI: 10.1016/j.compedu.2016.03.017

What the paper does: This paper explains how the Elo rating system, originally designed for chess, can be adapted for education by treating a student response as a match between a learner and an item. It surveys Elo variants, compares Elo to alternatives such as IRT and other learner models, and illustrates the approach in an adaptive geography practice system.

Why it is useful for us: This is the best practical starting point for an MVP. It supports online updates after every attempt, is simple to implement in a production backend, and directly estimates user ability and problem difficulty without requiring a heavy offline calibration pipeline at the beginning.

What we should take from it: Use Elo-style updates as the online core of the system. In our case, the basic idea becomes a multidimensional version where each problem updates several topic abilities at once. It is especially useful for bootstrapping a live recommender before enough data exists for more advanced models.

Main limitation for our use case: Standard Elo is not a full learning model. It tracks relative performance well, but it does not explicitly represent mastery transitions, forgetting, or concept prerequisites.

2. Vesin et al. 2022

Vesin, B., Mangaroska, K., Akhuseyinoglu, K., & Giannakos, M. (2022). Adaptive assessment and content recommendation in online programming courses: On the use of Elo-rating. ACM Transactions on Computing Education, 22(3), 1-27. DOI: 10.1145/3511886

What the paper does: This is the closest paper to our exact problem. It applies a modified Elo-rating approach in an online programming tutoring system to estimate learning content difficulty and student proficiency, then uses those estimates to recommend personalized programming content. The authors evaluate it over a semester and report that the recommendations were relevant and also useful for teacher insight.

Why it is useful for us: It validates that Elo is not only theoretically suitable but operationally useful in programming education. It gives us domain confidence that difficulty estimation and recommendation work differently in programming than in standard quiz systems, because programming exercises require concept transfer and problem-solving rather than isolated fact recall.

What we should take from it: Use an Elo-based student-content matching layer in the recommender and treat recommendation as a pairing problem between learner proficiency and exercise difficulty. This paper strongly supports the architecture we implemented in Perl: online updates plus recommendation from estimated fit.

Main limitation for our use case: The paper is much closer to course content recommendation than to large-scale competitive-problem recommendation. For a richer online judge, we still need better handling of multiple topic tags, repeated submissions, and uncertainty in problem metadata.

3. Bulut et al. 2023

Bulut, O., Shin, J., Yildirim-Erbasli, S. N., Gorgun, G., & Pardos, Z. A. (2023). An Introduction to Bayesian Knowledge Tracing with pyBKT. Psych, 5(3), 770-786. DOI: 10.3390/psych5030050

What the paper does: This is an accessible introduction to Bayesian Knowledge Tracing, including its parameters, assumptions, and practical implementation. BKT models whether a learner has mastered a skill and updates that mastery probability over time as the learner answers more items.

Why it is useful for us: BKT addresses the main thing Elo does not model well: learning progress over time at the skill level. If we want to know whether a learner is actually improving in recursion or graph traversal, rather than just whether they are stronger than the current item set, BKT gives a clear conceptual framework.

What we should take from it: Use BKT or a BKT-inspired mastery layer per topic or subtopic once we have clean event logs. This is particularly useful for sequencing problems, detecting weak concepts, and answering questions like “is the learner ready for the next DP problem family?”

Main limitation for our use case: Classical BKT assumes explicit knowledge components and often works best when each item maps cleanly to one skill. Algorithmic problems usually involve several skills at once, so we need either careful tagging or a more flexible extension.

4. Yang et al. 2025

Yang, L., Sun, X., Li, H., Xu, R., et al. (2025). Difficulty aware programming knowledge tracing via large language models. Scientific Reports, 15, 11475. DOI: 10.1038/s41598-025-96540-3

What the paper does: This paper extends programming knowledge tracing by making difficulty explicit. It distinguishes between two kinds of difficulty: knowledge-concept difficulty and problem text understanding difficulty. It uses LLM-generated difficulty annotations together with attention and graph-based modeling to improve prediction of student performance.

Why it is useful for us: This is highly relevant because algorithmic problems are difficult for more than one reason. Some are hard because the algorithmic idea is hard, while others are hard because the statement is long, tricky, or loaded with edge cases. Treating all difficulty as one number loses useful signal.

What we should take from it: Split problem difficulty into at least two components in our data model: algorithmic difficulty and statement-comprehension difficulty. Even if we do not use an LLM-heavy model initially, this paper gives a strong conceptual reason to keep those signals separate.

Main limitation for our use case: The full model is more complex than an MVP needs and depends on strong annotation quality. We should borrow the idea of multiple difficulty channels before borrowing the whole architecture.

5. Yera Toledo et al. 2018

Yera Toledo, R., Caballero Mota, Y., & Martínez, L. (2018). A Recommender System for Programming Online Judges Using Fuzzy Information Modeling. Informatics, 5(2), 17. DOI: 10.3390/informatics5020017

What the paper does: This paper proposes a recommender system specifically for programming online judges. The core idea is to model uncertainty in user behavior with fuzzy techniques, preprocess noisy data, compute user similarities, and then recommend problems accordingly. The authors report improvements over earlier POJ recommendation approaches, especially for short recommendation lists.

Why it is useful for us: It addresses the exact online judge context, where users can behave inconsistently, solve at different paces, and generate noisy interaction histories. That matters because submissions, retries, and problem selection in a POJ are messier than in a controlled course system.

What we should take from it: Add uncertainty-awareness to the recommendation layer. In practice, this means not overtrusting very sparse user histories, weighting recent data more carefully, and optionally blending content-fit with collaborative signals from similar users once we have enough data.

Main limitation for our use case: Fuzzy collaborative recommendation is useful, but by itself it is not enough for pedagogy. It can tell us what similar users did, but it does not directly model concept mastery or latent problem difficulty as cleanly as Elo or IRT-based approaches.

6. Bolsinova et al. 2025/2026

Bolsinova, M., Gergely, B., & Brinkhuis, M. J. S. (2026). Keeping Elo alive: Evaluating and improving measurement properties of learning systems based on Elo ratings. British Journal of Mathematical and Statistical Psychology, 79(1), 95-110. DOI: 10.1111/bmsp.12395

What the paper does: This paper studies the statistical behavior of Elo in educational settings and shows an important failure mode: when item selection is adaptive and item difficulties are updated together with student abilities, the rating variance can inflate artificially and the ratings may fail to converge. The authors propose a two-parallel-chain solution to reduce this feedback loop.

Why it is useful for us: It tells us where a simple MVP can go wrong. If we use one Elo loop both to estimate ratings and to choose the next problems, then the system can gradually distort its own measurements. That matters a lot in a recommendation platform because recommendations directly change the data we observe next.

What we should take from it: Keep Elo, but use guardrails. The safest practical version for our system is online Elo for short-term updates plus offline recalibration, and eventually some separation between selection logic and rating updates. This paper is the main reason not to treat Elo as a complete end-state solution.

Main limitation for our use case: It is a methodological warning paper rather than a full recommendation blueprint. Its value is in improving system design discipline and protecting us from biased measurements.

7. IRT Overview

Columbia University Mailman School of Public Health. Item Response Theory overview. Source: publichealth.columbia.edu

What the source does: This overview explains the core IRT concepts that matter most to our problem: latent ability, item difficulty, item discrimination, and how the probability of success changes as learner ability increases. It defines difficulty as the ability level where the item has roughly a 50% success probability and discrimination as how sharply the item separates stronger and weaker learners.

Why it is useful for us: IRT gives a rigorous language for the thing we are trying to estimate. It helps us define problem difficulty more precisely than acceptance rate, and it explains why some problems are better than others for measuring skill. Two problems can have the same acceptance rate but very different discrimination value.

What we should take from it: Use IRT concepts to guide the meaning of our rating fields, even if the online system starts with Elo. In particular, difficulty should represent a latent threshold, and eventually we should learn discrimination for problems that are especially informative in distinguishing learners near a given ability band.

Main limitation for our use case: Full IRT calibration is statistically stronger but operationally heavier than Elo. It is usually better as a batch recalibration layer than as the first thing we build online.

How the Papers Fit Together

These sources are complementary rather than competing. Pelanek 2016 and Vesin et al. 2022 support the practical online rating core. Bulut 2023 supports adding interpretable skill mastery tracking over time. Yang et al. 2025 motivates separating kinds of difficulty in programming problems. Yera Toledo et al. 2018 adds recommendation ideas specific to online judges and highlights uncertainty. Bolsinova et al. 2025/2026 is the cautionary paper that tells us how to avoid measuring ourselves into a biased corner. The IRT overview gives the formal measurement language behind difficulty and discrimination.

Recommended Design Direction

  • Use a multidimensional Elo-style update online for fast skill and difficulty adaptation.
  • Store multiple weighted skills per problem instead of one flat topic label.
  • Track at least two problem-difficulty components: algorithmic difficulty and statement-comprehension difficulty.
  • Add a mastery layer per skill later using BKT or a similar probabilistic tracing model.
  • Use collaborative or fuzzy recommendation only as a secondary signal, not the core pedagogical signal.
  • Run periodic recalibration so the recommendation loop does not bias the ratings over time.

Bottom Line

If we translate the literature into an implementation strategy, the cleanest path is: start with online Elo, structure the data as if an IRT or knowledge-tracing layer will arrive later, model problem difficulty more carefully than raw acceptance rate, and keep recalibration in the loop so the system stays statistically sane. That gives us something practical, explainable, and well aligned with the programming-education literature.

Notes on interpretation: some of the “what we should take from it” sections are design inferences drawn from the papers rather than direct claims made by the authors. The source URLs above are the main references used while preparing this report.
